1. 总体策略

折叠屏适配不要以“设备型号”作为判断依据，而应以当前应用窗口的可用宽度、分屏状态和内容关系来决定布局。官方规范布局将常见信息架构抽象成 List-Detail、Feed、Supporting Pane 等模式，本项目 demo 也按这些模式拆分。

辅助窗口 主内容与辅助内容单屏切换 Main + Supporting 并排，宽屏可加 Extra 让上下文资料靠近主任务
导航 底部导航 Navigation Rail 或宽导航栏 导航形态变化，业务 destination 不变

2. 断点与布局模型

当前 demo 使用的主要宽度阈值如下。Compose 侧依赖 WindowSizeClass 和 Material 3 Adaptive scaffold；View 侧用 view 实际宽度换算 dp 后自行切换布局。

宽度范围 推荐布局 Compose demo View demo
< 600dp Compact：单窗格、底部导航、列表/详情跳转 WindowSizeClass 小于 medium SlidingPaneLayout 锁定为单 pane 体验，列表点击滑入详情
≥ 600dp Medium：双窗格、双列 Feed、Navigation Rail WIDTH_DP_MEDIUM_LOWER_BOUND ListDetail 可由 SlidingPaneLayout 自动并排；Files/Supporting 切到横向布局
≥ 840dp Expanded：三列 Feed、更宽导航面 WIDTH_DP_EXPANDED_LOWER_BOUND Feed 三列，Navigation 宽模式
≥ 1200dp Large：更高密度网格，按业务谨慎增加窗格 Feed 四列 Feed 四列

关键原则是“布局可变，状态稳定”：选中项、当前 destination、详情内容和返回栈语义不能因为折叠、展开、分屏或窗口拖拽而丢失。

3.1 Compose 列表详情布局

官方 List-Detail 模式适合“左侧集合 + 右侧详情”的信息架构。当前 demo 使用 NavigableListDetailPaneScaffold 和 rememberListDetailPaneScaffoldNavigator，在 medium 以上初始化 List + Detail；expanded 以上通过 Extra pane 增加附加信息区。

Compact 初始只显示列表。点击列表项后导航到详情。返回键通过 scaffold navigator 回到列表。
Medium / Expanded 列表保持可见，详情在右侧更新。更宽窗口可显示 Extra pane，用于 metadata、操作或上下文。 
val windowAdaptiveInfo = currentWindowAdaptiveInfo(supportLargeAndXLargeWidth = true)
val showExtraPane = windowAdaptiveInfo.windowSizeClass.isWidthAtLeastBreakpoint(
    WindowSizeClass.WIDTH_DP_EXPANDED_LOWER_BOUND
)
val baseScaffoldDirective =
    calculatePaneScaffoldDirectiveWithTwoPanesOnMediumWidth(windowAdaptiveInfo)
val scaffoldDirective = if (showExtraPane) {
    baseScaffoldDirective.copy(maxHorizontalPartitions = 3)
} else {
    baseScaffoldDirective
}

val navigator = rememberListDetailPaneScaffoldNavigator<Int>(
    scaffoldDirective = scaffoldDirective,
    initialDestinationHistory = initialHistory,
)

NavigableListDetailPaneScaffold(
    navigator = navigator,
    listPane = { AnimatedPane { MailListPane(...) } },
    detailPane = { AnimatedPane { MailDetailPane(...) } },
    extraPane = { AnimatedPane { MailExtraPane(...) } },
)

实现要点

  • selectedId 是业务状态，pane 切换时不能重置。
  • initialDestinationHistory 决定初始展开形态，避免大屏打开后还停留在单 pane。
  • LaunchedEffect(showListAndDetail, showExtraPane) 处理窗口大小变化后的 pane 纠偏。
  • 列表项点击后只更新 selectedId 并导航到 Detail，不创建新的页面模型。

3.2 Compose Feeds 布局

Feed 是同级内容集合，展开后不应该变成“列表详情”，而应提高横向内容密度。当前 demo 使用 LazyVerticalGrid，按窗口宽度决定列数，重点卡片可跨 2 列。 

private fun WindowSizeClass.feedColumnCount(): Int = when {
    isWidthAtLeastBreakpoint(WIDTH_DP_XLARGE_LOWER_BOUND) -> 4
    isWidthAtLeastBreakpoint(WIDTH_DP_LARGE_LOWER_BOUND) -> 4
    isWidthAtLeastBreakpoint(WindowSizeClass.WIDTH_DP_EXPANDED_LOWER_BOUND) -> 3
    isWidthAtLeastBreakpoint(WindowSizeClass.WIDTH_DP_MEDIUM_LOWER_BOUND) -> 2
    else -> 1
}

LazyVerticalGrid(columns = GridCells.Fixed(columns)) {
    item(span = { GridItemSpan(columns) }) {
        FeedHeader(columns = columns)
    }
    items(
        items = adaptiveFeedItems,
        key = { it.id },
        span = { item -> GridItemSpan(if (item.featured) minOf(2, columns) else 1) },
    ) { item ->
        FeedCard(item = item, expanded = item.featured && columns > 1)
    }
}

实现要点

  • Compact 保持单列，保证扫描和滚动稳定。
  • Medium 使用两列，适合折叠屏展开后的自然宽度。
  • Expanded 及以上使用三到四列，但卡片最大宽度仍应可读。
  • 重点卡片跨列只影响视觉层级，不改变数据顺序。

3.3 Compose Files 布局

Files 类型界面通常包含分类入口、最近文件、清理建议和工具动作。当前 demo 在 compact 下使用单列滚动；medium 以上切成左右两列：左侧是主要文件内容，右侧承载清理建议。

        item { FilesHeader() }
        item { QuickActionRow() }

实现要点

  • 主要任务是“找文件”，清理和工具属于辅助信息，大屏适合右侧独立承载。
  • 分类网格保持固定 4 列，减少展开后视觉跳动。
  • 左右两列都用 LazyColumn，避免一个长列表拖累另一侧阅读。

3.4 Compose 辅助窗口布局

Supporting Pane 适合主内容旁边放“强相关但非主流程”的信息，例如笔记、目录、引用、属性和工具。当前 demo 使用 SupportingPaneScaffold，在可用宽度允许时同时显示 Main 和 Supporting，并为更宽窗口预留 Extra。 

val scaffoldDirective =
    calculatePaneScaffoldDirectiveWithTwoPanesOnMediumWidth(windowAdaptiveInfo)
val showSupportingPane = scaffoldDirective.maxHorizontalPartitions >= 2
val showExtraPane = scaffoldDirective.maxHorizontalPartitions >= 3

val navigator = rememberSupportingPaneScaffoldNavigator<String>(
    scaffoldDirective = scaffoldDirective,
    initialDestinationHistory = initialHistory,
)

SupportingPaneScaffold(
    directive = navigator.scaffoldDirective,
    scaffoldState = navigator.scaffoldState,
    mainPane = { AnimatedPane { DocumentPane(...) } },
    supportingPane = { AnimatedPane { ReadingSupportPane(...) } },
    extraPane = { AnimatedPane { ReadingExtraPane(...) } },
)

实现要点

  • Compact 下 Main 是默认入口，用户主动打开 Supporting，返回后回到 Main。
  • 展开时 Supporting 常驻，选择目录项时只更新主内容状态。
  • Extra pane 适合放全局工具或低频设置，不要抢主内容宽度。

3.5 Compose 自适应导航

自适应导航的重点是“同一套 destination 模型，不同窗口使用不同导航组件”。当前 demo 使用 NavigationSuiteScaffold，由官方组件根据窗口宽度选择底部导航、Navigation Rail 或更宽的导航形态。 

var selectedDestination by remember {
    mutableStateOf(AdaptiveNavigationDestination.Home)
}

NavigationSuiteScaffold(
    navigationSuiteItems = {
        AdaptiveNavigationDestination.entries.forEach { destination ->
            item(
                selected = destination == selectedDestination,
                onClick = { selectedDestination = destination },
                icon = { DestinationIcon(destination) },
                label = { Text(destination.label) },
            )
        }
    },
) {
    AdaptiveNavigationContent(
        destination = selectedDestination,
        windowSummary = windowAdaptiveInfo.windowSizeClass.widthDescription(),
    )
}

实现要点

  • 业务只关心 selectedDestination，不根据宽度手写三套导航。
  • 窗口变化不应重置当前 tab。
  • 内容区域按照 destination 渲染，导航 chrome 由 scaffold 接管。

4. Android 原生 View 实现方案

Android View demo 位于 /Users/onuszhao/Code_KMM/KmmDemoTest 项目下，以 FoldableDemoActivity 作为原生适配入口。它不依赖 Compose scaffold，而是使用 FrameLayout、LinearLayout、RecyclerView、GridLayoutManager、SlidingPaneLayout 和 Activity Embedding 规则实现自适应。

View 侧统一判断方式

private fun widthDp(): Int {
    val w = if (width > 0) width else resources.displayMetrics.widthPixels
    return (w / resources.displayMetrics.density).toInt()
}

View 方案适合现有原生页面低成本改造：不用整体迁移 Compose。对于列表详情场景，优先使用 SlidingPaneLayout 让系统组件处理折叠、并排、滑动和返回语义；对于 Feed、Files、辅助窗口等更自由的界面，再按窗口宽度手动切换布局。

4.1 原生容器 Activity

FoldableDemoActivity 是原生 demo 的统一入口。顶部横向滚动 tab 选择 demo，下面的 FrameLayout 作为内容容器。Activity Embedding 是独立 Activity 组，点击 tab 时直接启动 primary Activity。 

private fun renderDemo(type: FoldableDemoType) {
    contentContainer.removeAllViews()
    val view = when (type) {
        FoldableDemoType.LIST_DETAIL -> ListDetailDemoView(this)
        FoldableDemoType.FEED -> FeedDemoView(this)
        FoldableDemoType.FILES -> FilesDemoView(this)
        FoldableDemoType.SUPPORTING_PANE -> SupportingPaneDemoView(this)
        FoldableDemoType.ADAPTIVE_NAVIGATION -> AdaptiveNavigationDemoView(this)
        FoldableDemoType.ACTIVITY_EMBEDDING -> error("Handled above")
    }
    currentBackHandler = view as? FoldableBackHandler
    contentContainer.addView(view)
}

实现要点

  • selectedType 通过 onSaveInstanceState 保存，旋转或重建后仍停留在当前 demo。
  • 实现 FoldableBackHandler 的子 View 可以优先消费返回事件。
  • 所有 demo 直接创建 View，便于说明原生适配逻辑，但生产项目可以迁移到 XML、Fragment 或自定义 ViewGroup。

4.2 原生列表详情布局

原生列表详情现在推荐使用 SlidingPaneLayout 实现。它是 Android View 体系里专门面向双窗格和可折叠列表详情的容器：子 View 按“列表 pane + 详情 pane”声明，系统根据可用宽度决定是并排显示，还是在紧凑宽度下让详情页滑入覆盖列表。

这里的“折叠/展开”不是由业务代码直接判断设备类型，而是由 SlidingPaneLayout 在测量阶段决定：如果父容器宽度足够容纳两个 pane，组件进入非滑动的并排态；如果宽度不足，组件进入可滑动的折叠态。业务代码只需要根据 isSlideable 和 isOpen 处理交互。

紧凑宽度 isSlideable == true。默认只看列表。点击列表项后调用 openPane() 打开详情；返回时先调用 closePane() 回到列表。
展开宽度 isSlideable == false。列表和详情同时可见。点击列表项只更新详情内容，不需要重建整个布局，也不需要手动处理横向分栏。 
private lateinit var slidingPaneLayout: SlidingPaneLayout
private var selectedItem = sampleMails().first()

private fun createListDetailWithSlidingPane(): View {
    slidingPaneLayout = SlidingPaneLayout(this).apply {
        layoutParams = ViewGroup.LayoutParams(MATCH_PARENT, MATCH_PARENT)
        lockMode = SlidingPaneLayout.LOCK_MODE_UNLOCKED
    }

    val listPane = RecyclerView(this).apply {
        layoutManager = LinearLayoutManager(this@FoldableDemoActivity)
        adapter = SimpleTextAdapter(sampleMails(), { selectedItem.id }) { item ->
            selectedItem = item
            renderDetail(item)
            if (slidingPaneLayout.isSlideable) {
                slidingPaneLayout.openPane()
            }
        }
    }

    val detailPane = createDetailPane(selectedItem)

    slidingPaneLayout.addView(
        listPane,
        SlidingPaneLayout.LayoutParams(dp(340), MATCH_PARENT),
    )
    slidingPaneLayout.addView(
        detailPane,
        SlidingPaneLayout.LayoutParams(MATCH_PARENT, MATCH_PARENT),
    )
    return slidingPaneLayout
}

返回处理

onBackPressedDispatcher.addCallback(this) {
    val shouldCloseDetail = ::slidingPaneLayout.isInitialized
        && slidingPaneLayout.isSlideable
        && slidingPaneLayout.isOpen

    if (shouldCloseDetail) {
        slidingPaneLayout.closePane()
    } else {
        isEnabled = false
        onBackPressedDispatcher.onBackPressed()
    }
}

这个实现的核心是让 SlidingPaneLayout 管理“是否能并排”和“是否需要滑动打开详情”，业务层只维护 selectedItem 和详情内容。相比手写 widthDp >= 600 的 LinearLayout 方案，它减少了重复布局分支，也更贴近官方双窗格方案。

4.3 SlidingPaneLayout 方案细节

SlidingPaneLayout 的适配逻辑可以理解为“一个组件承载两套交互”：窗口足够宽时是固定双窗格，窗口不足时是列表到详情的滑动导航。它特别适合邮件、设置、文件目录、联系人、会话列表等标准 List-Detail 页面。

状态控制责任

状态/动作 谁负责 具体逻辑
是否折叠 SlidingPaneLayout 组件在测量子 View 后判断宽度是否足够。足够则 isSlideable=false，列表和详情并排；不足则 isSlideable=true，进入滑动折叠模式。
详情是否打开 SlidingPaneLayout isOpen 表示滑动 pane 当前是否打开。紧凑态点击列表后调用 openPane()；返回时调用 closePane()。
选中哪条数据 业务层 Activity/ViewModel 保存 selectedItem 或 selectedItemId。折叠、展开或 pane 开关都不应该重置该状态。
列表点击行为 业务层 + 组件 业务层更新详情内容；如果 isSlideable 为 true，再调用 openPane()。展开并排态无需打开 pane。
返回键行为 业务层 仅当 isSlideable && isOpen 时关闭详情 pane；否则交给 Activity 原有返回栈。
窗口尺寸变化 SlidingPaneLayout 窗口折叠、展开、分屏拖拽后，组件重新测量并更新 isSlideable。业务层不需要销毁并重建列表详情布局。

折叠态和展开态判断

private fun updateListDetailUi() {
    val compactSlidingMode = slidingPaneLayout.isSlideable
    val detailVisibleInCompact = slidingPaneLayout.isSlideable && slidingPaneLayout.isOpen

    if (compactSlidingMode) {
        // 折叠态：一次只聚焦一个 pane。
        // 可以显示详情页返回按钮，或让系统返回键关闭详情 pane。
        detailBackButton.isVisible = detailVisibleInCompact
    } else {
        // 展开态：列表和详情同时可见。
        // 不展示详情页返回按钮，列表点击只刷新详情内容。
        detailBackButton.isVisible = false
    }
}
职责 推荐处理
列表 pane 使用固定或最小宽度，例如 320dp 到 360dp；承载 RecyclerView、筛选和列表选中态。
详情 pane 使用剩余空间；紧凑屏下作为滑入页面；展开屏下与列表同时可见。
选中状态 放在 Activity/ViewModel 层，例如 selectedItemId，不要绑定到具体 pane 是否打开。
返回语义 如果详情 pane 已打开，先关闭 pane；否则退出 Activity 或交给上层导航。
窗口变化 不要手动销毁列表详情结构；让 SlidingPaneLayout 根据测量结果切换并排或滑动。

4.4 原生 Feeds 布局

FeedDemoView 使用 RecyclerView + GridLayoutManager 实现动态列数。相比 Compose 的 LazyVerticalGrid，原生版本需要手动设置 spanCount 并通过 SpanSizeLookup 控制重点卡片跨列。 

override fun onSizeChanged(w: Int, h: Int, oldw: Int, oldh: Int) {
    val widthDp = (w / resources.displayMetrics.density).toInt()
    gridLayoutManager.spanCount = when {
        widthDp >= 1200 -> 4
        widthDp >= 840 -> 3
        widthDp >= 600 -> 2
        else -> 1
    }
    gridLayoutManager.spanSizeLookup.invalidateSpanIndexCache()
    feedAdapter.notifyDataSetChanged()
}
gridLayoutManager.spanSizeLookup = object : GridLayoutManager.SpanSizeLookup() {
    override fun getSpanSize(position: Int): Int =
        feedAdapter.spanSize(position, gridLayoutManager.spanCount)
}

实现要点

  • 列数变化只改变布局密度，不改变 adapter 数据。
  • 首条或重点内容可以跨全部列或 2 列，增强层级。
  • 窗口尺寸变化后需要刷新 span cache 和 adapter。

4.5 原生 Files 布局

FilesDemoView 在紧凑屏下使用一个 ScrollView 串起标题、快捷操作、分类、最近文件和清理卡片。展开后根 LinearLayout 改为横向，清理卡片移动到右侧独立列。 

val expanded = widthDp() >= 600
val root = LinearLayout(context).apply {
    orientation = if (expanded) LinearLayout.HORIZONTAL else LinearLayout.VERTICAL
}

root.addView(main, LinearLayout.LayoutParams(0, -1, if (expanded) 1f else 0f))
if (expanded) {
    root.addView(scrollColumn {
        setPadding(context.dp(16), context.dp(56), 0, 0)
        addSpaced(cleanCard())
    }, LinearLayout.LayoutParams(0, -1, 1f))
}

实现要点

  • 文件分类用 GridLayout 固定 4 列，展开后保持认知位置。
  • 右侧清理区只在大屏常驻，紧凑屏保持在主滚动流末尾。
  • 窗口宽度变化时调用 render() 重建布局。

4.6 原生辅助窗口布局

SupportingPaneDemoView 手动模拟 Compose SupportingPaneScaffold。紧凑屏只显示 Main 或 Supporting；展开后同时显示 Main 和 Supporting；宽度达到约 1000dp 后再增加 Extra 工具栏。 

if (expanded) {
    root.addView(documentPane(false), LinearLayout.LayoutParams(0, -1, 2f))
    root.addView(supportPane(false), LinearLayout.LayoutParams(0, -1, 1f))
    if (widthDp() >= 1000) {
        root.addView(extraPane(), LinearLayout.LayoutParams(0, -1, 1f))
    }
} else {
    root.addView(
        if (showingSupportCompact) supportPane(true) else documentPane(true),
        LinearLayout.LayoutParams(-1, -1),
    )
}

实现要点

  • showingSupportCompact 表示紧凑屏当前是否在辅助页。
  • 实现 FoldableBackHandler，紧凑屏返回优先回 Main。
  • 展开屏不显示 Back 按钮，因为 Main 和 Supporting 同时可见。

4.7 原生自适应导航

AdaptiveNavigationDemoView 手动实现三种导航形态：紧凑底部导航、medium 侧边 rail、expanded 宽导航面。业务 destination 枚举保持不变。 

val mode = when {
    widthDp >= 840 -> NavMode.WIDE
    widthDp >= 600 -> NavMode.RAIL
    else -> NavMode.BOTTOM
}

if (mode == NavMode.BOTTOM) {
    container.addView(contentView(), LinearLayout.LayoutParams(-1, 0, 1f))
    container.addView(navBar(horizontal = true), LinearLayout.LayoutParams(-1, context.dp(64)))
} else {
    val navWidth = if (mode == NavMode.WIDE) context.dp(220) else context.dp(84)
    container.addView(navBar(horizontal = false), LinearLayout.LayoutParams(navWidth, -1))
    container.addView(contentView(), LinearLayout.LayoutParams(0, -1, 1f))
}

实现要点

  • selectedDestination 是稳定业务状态，切换导航模式不重置。
  • 底部导航占用高度，rail/宽导航占用宽度，内容区域按剩余空间渲染。
  • 生产实现建议使用 Material Components 的 NavigationBarView / NavigationRailView 统一样式。

4.8 Activity Embedding

Activity Embedding 适合既有多 Activity 架构：不把页面合并成单 Activity，也能在大屏上把 primary 和 secondary 并排展示。当前 demo 通过 AndroidX Window 的 RuleController 加载 XML 规则。

初始化规则

class FoldableSplitInitializer : Initializer<Unit> {
    override fun create(context: Context) {
        val controllerClass = Class.forName("androidx.window.embedding.RuleController")
        val getInstance = controllerClass.getMethod("getInstance", Context::class.java)
        val parseRules = controllerClass.getMethod("parseRules", Context::class.java, Int::class.javaPrimitiveType)
        val setRules = controllerClass.getMethod("setRules", Set::class.java)
        val controller = getInstance.invoke(null, context)
        val rules = parseRules.invoke(null, context, R.xml.foldable_activity_embedding_rules)
        setRules.invoke(controller, rules)
    }
}

SplitPairRule

<SplitPairRule
    window:clearTop="true"
    window:finishPrimaryWithSecondary="never"
    window:finishSecondaryWithPrimary="always"
    window:splitMinWidthDp="600"
    window:splitRatio="0.35">
    <SplitPairFilter
        window:primaryActivityName=".feature.foldable.embedding.FoldableEmbeddingPrimaryActivity"
        window:secondaryActivityName=".feature.foldable.embedding.FoldableEmbeddingSecondaryActivity" />
</SplitPairRule>

业务流程

  1. FoldableDemoActivity 点击 ActivityEmbedding tab，启动 FoldableEmbeddingPrimaryActivity。
  2. Primary Activity 显示列表，点击项后启动 FoldableEmbeddingSecondaryActivity 并通过 intent 传详情数据。
  3. 当窗口宽度满足 splitMinWidthDp=600 且系统支持 embedding 时，secondary 出现在右侧；否则作为普通 Activity 全屏打开。
  4. finishSecondaryWithPrimary="always" 保证 primary 关闭时 secondary 一起结束；finishPrimaryWithSecondary="never" 允许关闭 secondary 后保留 primary。
当前 app 依赖中包含 androidx.window:window:1.5.1 和 androidx.startup:startup-runtime:1.2.0，满足 Activity Embedding 规则加载所需基础能力。
